# 系统现状与技术债清单

更新时间：2026-07-12

## 职责边界

- Python 负责公开信息采集、正文/附件字段提取、去重、增量哈希、pending 队列生成和 JSON 持久化。
- Codex 模型负责正式的岗位语义理解、个人条件匹配、`match / possible / no` 判断、理由、冲突和原文证据。
- Python 不应以规则引擎代替 Codex 完成正式语义判断。
- 例外：岗位缺少 AI 缓存时，前端目前会执行一套简化正则兜底；该结果不是正式 Codex 分析。

## P0：输入与判断一致性

1. **公告正文没有完整进入模型输入。** 自动化提示词描述的输入比 `ai-pending.json` 实际包含的内容更丰富。
2. **北京人社局公告正文变化未必触发重新分析。** 准备哈希没有包含公告级正文哈希、完整正文和全部附件元数据。
3. **`score` 没有正式评分公式。** 当前分数是历史模型输出形成的档位约定，无法稳定复现。
4. **前端不校验 AI 缓存的 `content_hash`。** 页面只按岗位 ID 读取结果，正确性依赖准备脚本和发布流程。
5. **个人档案与前端兜底规则硬编码。** AI 使用 `data/profile.json`，但页面条件标签和兜底正则没有动态读取该档案。

## P1：展示与数据质量

6. **`no` 岗位完全隐藏。** 用户无法临时查看或复核被排除岗位，应增加“显示不符合岗位”开关。
7. **历史缓存影响统计。** 当前岗位 851 条，AI 缓存 2,056 条，其中 1,205 条是已不在当前采集集中的历史结果；页面的“Codex 已分析”数量可能产生误解。
8. **部分缓存缺少 evidence。** 当前缓存共 152 条没有 evidence，其中 125 条仍属于当前岗位，不完全满足现行分析规范。
9. **截止时间缺失较多。** 当前岗位中 577 条标准化截止时间为空，页面只能依赖发布时间或最后验证时间的 30 天新鲜度规则。

## P2：采集覆盖和字段质量

10. **小米职责字段疑似对调。** 当前适配器可能把工作职责写入 `requirements`，把工作要求写入 `responsibilities`。
11. **小米不是全量发现。** 目前只检查代码中固定的 5 个岗位 ID，无法发现新增设计岗位。
12. **通用来源存在采集预算上限。** 默认只深入读取前 10 个详情、最多解析 3 个 Excel 附件，可能漏掉岗位。

## P2：验证体系

13. **前端渲染测试已经过时。** `tests/rendered-html.test.mjs` 仍检查早期 starter skeleton，与当前招聘页面不一致；需要重写为当前页面的数据加载、筛选、隐藏和展示测试。

## 后续处理原则

- 优先补齐模型输入和哈希链路，再调整评分或 UI，避免在输入不完整时优化表面结果。
- 修复后增加对应测试，并用真实公告、附件岗位和内容变更样例验证。
- 对历史缓存进行迁移或清理时，必须保留可恢复备份，并重新生成 pending 验证覆盖率。

## 已确认的采集器演进方向

- 不把通用 Python 表头别名当作完整语义理解；逐个网站建立来源专用字段映射和采集测试。
- 通用 Excel 解析必须保留 `raw_fields`，未知列不得静默丢弃，供后续 Codex 或人工复核。
- 取消跨栏目按文章编号推断“同一公告”的语义型去重；仅保留完全相同 URL 或完全相同岗位 ID 的技术型去重，避免页面键冲突和 AI 缓存互相覆盖。
